M3 - Threads

Threads — Motivation

  1. Let’s a single process make use of multiple cores.
  1. Separate processes need pipes to communicate with each other. This is wasteful if many seperate P’s need access to the same data. It would be good if multiple tasks could run and share/modify some of the same data in memory.

Benefits of Threads

  1. Allows 1 process to use multiple cores
  1. Faster than setting up a new process (less context to save)
  1. A program can do I/O and computation at the same time.
  1. Special apps can run on a kernel thread and don’t need to transfer control to do syscall’s

syscall’s for creating processes (ie: done in kernel-space)

Example
// Signature of Function to run on thread(s)
void * print_message( void *ptr );

main() {
   pthread_t thread1, thread2;
   char *message1 = "Thread 1"; //
   char *message2 = "Thread 2";
   int  iret1, iret2; //return values of the threads

  /* Create 2 threads */

   iret1 = pthread_create(&thread1, NULL, print_message, (void*) message1);
   iret2 = pthread_create(&thread2, NULL, print_message, (void*) message2);

   /* Wait until threads are complete before main continues. Unless we
		  wait we run the risk of executing an exit which will terminate
		  the process and all threads before the threads have completed. */

   pthread_join(thread1, NULL);
   pthread_join(thread2, NULL); 

   printf("Thread 1 returns: %d\n", iret1);
   printf("Thread 2 returns: %d\n", iret2);

   exit(0);
}

void *print_message(void * ptr) {
     char *message;
     message = (char *) ptr;
     printf("%s \n", message);
}

output: 

Thread 1
Thread 2
Thread 1 returns: 0
Thread 2 returns: 0

Models

Only Kernel Threads

Limitations

  • slow: every thread operation is done in the kernel (lots of syscall’s)
  • memory cost: requires a stack within the kernel and within user-space (TLB)

1 kernel-thread, nnn﻿ user-thread’s

  • Implementation
    1. User space library holds a queue of user-threads. It decides how to schedule them.
    1. Replace blocking syscall’s

      blocking syscall

      e.g: read, write

      Blocks the KT until data is available ⇒ KT → wait state ⇒ no UT can run

      Non-Blocking syscall

      • If file/data is not available, return nothing
      • It’s up to the user-space to switch the running thread.
    1. Schedule periodic timer signal: setitimer
      • When the timer fires, kernel switches the user-thread.

Limitations

  • many user threads for only 1 core
  • Blocking syscall can block all threads
  • Page fault blocks all threads
    • ie: touching memory that the kernel needs to fetch from disk ⇒ blocking syscall
  • Deadlock can happen easily

nnn﻿ user-threads, mmm﻿ kernel-threads

  • A blocking operations won’t block all UT’s at once.
    • Ideally want at least 1 KT per core
  • Threads are created & scheduled by the USTL.
    • thread_create, thread_exit still library functions
    • Determines which UT runs on which KT

Benifits

  • Harder to block all user-level threads at once
  • deadlocks are less likely (takes more user level threads making blocking syscall’s)

Limitations

  • Interaction between kernel and user space:
    • ULTL does not know how many KT’s are available (some could be blocked)
    • Kernel does not how importance of UT’s (preemption might cause huge delay)

Summary

  • Threads are best implemented as a library
  • Use KT’s if I/O concurrency is your main goal (lot’s of blocking calls)
  • Use n:mn:mn:m﻿ threading for highly concurrent with many thread switches
  • Threads introduce problems regarding concurrency/synchronization
