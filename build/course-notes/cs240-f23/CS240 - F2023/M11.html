Dictionaries for Integers

Hashing:

Was O(1)O(1)O(1)﻿ expected for all run-time, but this assumed that α\alphaα﻿ was kept small

We needed to re-hash; transfer everything to a new table…

This means loading all n/Bn/Bn/B﻿ block transfers

Very very slow… (PROBLEM)

Goal:

DS for hash-values that uses O(1)O(1) O(1)﻿ block transfers & never needs to load all blocks

Idea:

view hash-values as fixed-length bitstrings

Store trie of bit-strings whose leaves are blocks of memory

Trie of Blocks

Assume:

that the blocks store fixed-length bitstrings

not distinct (come from hash-values)

Build trie DDD﻿ (directory) of bit-strings

  • Stored in internal memory
  • Stop splitting in DDD﻿ when remaining items fit in one block
    • pruned trie but even earlier
  • Leaf has reference to a block
  • blocks store KVPs

Note: don’t care what happens in the block,just trying to find it.

search(k)

O(1)O(1)O(1)﻿

  • follow tree to leaf lll﻿
  • load block at lll﻿ (1 BLOCK TRANSFER)
  • search for kkk﻿ in lll﻿ (internal mem)

delete(k)

O(1)O(1)O(1)﻿

  • search(k) (1)
  • delete kkk﻿ from block
  • transfer updated block back (1)

insert(k)

O(1)O(1)O(1)﻿ expected

  • search(k) (1)
  • if block full: split while they have the same next bit
  • insert kkk﻿ into block
  • transfer updated block back (1)

External hashing discussion

  • Hashing collisions (= duplicate keys) are resolved within the block
  • Directory D typically fits into in internal memory.
  • Only O(1) block transfers expected for any operation
  • To make more space, we only add one block → We rarely change the size of the directory → We never have to move all items.
