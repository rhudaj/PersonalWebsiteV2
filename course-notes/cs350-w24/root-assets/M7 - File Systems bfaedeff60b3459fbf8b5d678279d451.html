syscallParams?
openpathOpen the file (creates a new one if it DNE). Return fd
closeinvalidate the fd
readfd, vaddr, lengthcopies data from file (@ file-position) → virtual address space
writefd, data, lengthcopies data from virtual address space → file (@ file-position)
seekfd, posChanges the file position (ie: for subsequent read/write)
  • EX, Sequential Read
    char buf[512];
int f = open("myfile", O_RDONLY);
for(int i=0; i<100; i++) {
	read(f,(void*)buf, 512);  // read 512 bytes from f to buff
}
close(f);
  • EX, Read w’ Seek (opposite direction)
    char buf[512];
int f = open("myfile", O_RDONLY);
for(int i=0; i<100; i++) {
	lseek(f, (100-i)*512, SEEK_SET); // seek position from end
	read(f,(void*)buf, 512);         
}
close(f);

File Systems

Create a Hard Link

open(/foo/misc/biz)
  • creates a new file if /foo/misc/biz DNE
  • creates a hard link in the directory /foo/misc

Changing Links 

link(/docs/a.txt, /foo/myA) // new hard link, myA in directory /foo
// the link refers to the i-number of /docs/a.txt, which must exist

Unlinking

unlink(/docs/b.doc) // remove hard-link btwn docs and b.doc

Multiple File Systems

More than 1 file system can be used at the same time (as different namespaces)

  • e.g, USB stick
  • e.g, on Windows, each FS has a unique namespace (ie: C) Paths are written with the start of this name: C:\user\cs350
  • Mount()

    Mounts a the FS somewhere under the root file system.

File System Design

Persistent

Data structure held on disk:

  • i-nodes (File metadata)
  • bitmaps (FS metadata)
  • data blocks (File Data)

Non-persistent

processes open files, stored in-memory

  • Open files
  • File Positions
  • Cached copies i-nodes
  • Cached copies data blocks
Persistent
Disk FS, Implementation (V1)

Setup

Want convenient chunks of sectors to work with that aligns with pages in memory.

Metadata

  • i-node: blocks holding metadata for multiple files (ie: what data blocks?)
    • Array of i-nodes are indexed with i-numbers
  • Superblock: metadata for the entire file system
  • Bitmaps: which i-nodes & blocks are unused

General Case

  • Most files are small
  • A few big files use most of the space
  • File systems contain lots of files
Non-Persistent
  • Stored in memory
  • FD-Table per process so that when P terminates, we can close them.
  • Process FD-Table have have pointers to entries to the open-file table.
  • FD’s are process specific (fd0fd_0fd0​﻿ in P1P_1P1​﻿ does not necessarily refer to the same file as fd0fd_0fd0​﻿ in P2P_2P2​﻿).
  • But, possible to have 2 P’s having the same file open:
    • fork : copies fd’s over (2 P’s have pointers to the same entry).
    • dup2 : duplicates an fd
  • System Wide Open-File Table
    • i-node cache: in-memory copies of recently used i-nodes
    • block cache: in-memory copies of recently used data blocks
    • read from cache if it’s there, otherwise pull from disk.
    • read(fd) updates the caches. write(fd) invalidates it.
Reading from a File

Creating & Writing to a File
  • writes are block level, 1 block contains many i-nodes ⇒ read block first, update desired i-node and write block back.
  • creating a file without a write ⇒ just an i-node (no data blocks or direct pointers)
  • when you write to a file, you allocate blocks
Deleting a File
  • remove entry from directory
  • remove file index (i-node) from i-node table
  • mark file’s data blocks free in free space index
