23 - Accessibility

Access to user interfaces is considered a human right

Goal: accessible interfaces for everyone

Types of disabilities

  • permanent (1 arm)
  • temporal (broken arm)
  • situational (ear infection, Driving a car, walking)

situational walking impairment

Problems:

  • requires larger target sizes
  • slower reading
  • gain less info

Solution:

  • bigger cues
  • less info

Age-Related Impairments

  • Keep information simple (cognitive)
  • High contrast colours, large text and icons (vision)
  • Large widget and button sizes (motor)

Chronic and Long-term Disabilities

  • 10 to 20% of population
  • OS already has support
  • Visual, Hearing & Cognitive Impairment

Accessibility Tree

  • The browser generates an accessibility tree from the DOM with accessibility-related information for most HTML elements
  • It can be viewed
  • DevTools Accessibility Tab

Tips

  • Include alt text for information images
    <p><img src="phone.png" alt="Telephone:"/>0123 456 7890</p>
  • Use headings, and use them correctly
  • Give links unique and descriptive names
  • Use colour with care
  • Use tables for tabular data, not for layout
  • Make dynamic content accessible
  • Make all content accessible using keyboard too

Semantic HTML

  • easier to navigate for visually impaired
  • does not use div or span
  • Avoid <div> for component root if semantic element possible
  • Use <Fragment> instead of <div> if root has no semantic purpose

Skip Link

  • Add link at top of page so screen readers can "skip to content‚Äù

ARIA Attributes

  • add semantics to elements
  • ARIA attributes only change the accessibility tree
  • ARIA Roles; Use ONLY if not possible to use an HTML semantic element

Testing Methods Disconnect your mouse and try to use your app

  1. Tab and Shift+Tab to focus element
  1. Enter to activate element
  1. Arrow keys when needed (e.g. menu, dropdown)
  1. Test with a screenreader
